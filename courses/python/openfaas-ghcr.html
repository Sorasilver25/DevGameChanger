OpenFaaS - Fonctions Python, packaging Docker et GHCR

Ce cours présente OpenFaaS comme une couche serverless légère pour exécuter des fonctions Python, puis montre comment emballer ces fonctions en image Docker, les publier dans GHCR et les déployer proprement.

Dernière mise à jour : 09.04.2026

1. Introduire OpenFaaS

OpenFaaS permet d'exposer des fonctions sous forme d'API HTTP sans gérer directement l'infrastructure d'exécution. L'idée est simple : vous écrivez une fonction, vous la packez dans une image, puis vous la déployez derrière le gateway OpenFaaS.

  • Déploiement rapide de fonctions Python, Node.js, Go ou autre
  • Scalabilité par fonction
  • Isolation via conteneurs
  • Observabilité simple grâce au gateway et aux logs
  • Intégration GitOps / CI-CD via images versionnées

Le pattern à retenir : une fonction = un contrat HTTP + une image Docker + une version explicite.

Fonction Python d'exemple : pretty print JSON

Voici une fonction simple et utile : recevoir un payload JSON et le renvoyer en version formatée. C'est typiquement le genre de fonction qu'on garde comme brique réutilisable dans OpenFaaS. 

import json


def handle(event, context):
    raw_body = event.body or "{}"
    try:
        payload = json.loads(raw_body)
    except json.JSONDecodeError:
        return {
            "statusCode": 400,
            "body": json.dumps({"error": "Payload JSON invalide"}, ensure_ascii=False),
            "headers": {"Content-Type": "application/json; charset=utf-8"},
        }

    pretty = json.dumps(payload, indent=2, ensure_ascii=False, sort_keys=True)
    return {
        "statusCode": 200,
        "body": pretty,
        "headers": {"Content-Type": "application/json; charset=utf-8"},
    }

Cette fonction est volontairement simple : elle montre comment standardiser une logique métier sous forme de fonction packagée.

2. Créer son package Docker pour OpenFaaS

Dans OpenFaaS, le packaging moderne passe généralement par une image Docker. Pour Python, la voie la plus propre consiste à partir d'un template OpenFaaS, puis à versionner l'image dans un registre comme GHCR. 

# Installer le CLI OpenFaaS
curl -sSL https://cli.openfaas.com | sudo sh

# Récupérer le template Python
faas-cli template store pull python3-http

# Créer une nouvelle fonction
faas-cli new pretty-json --lang python3-http

Ensuite, vous placez votre logique dans handler.py et vous gardez une version claire dans le fichier de fonction. 

version: 1.0
provider:
  name: openfaas
  gateway: http://127.0.0.1:8080
functions:
  pretty-json:
    lang: python3-http
    handler: ./pretty-json
    image: ghcr.io/votre-compte/openfaas-pretty-json:1.0.0

Publier dans GHCR

GHCR est un bon choix ici parce qu'il s'intègre naturellement à GitHub et permet de versionner les images comme des packages. Vous avez deux cas d'usage : publier manuellement ou via GitHub Actions.

Authentification manuelle :

export GHCR_USER="votre-compte"
export GHCR_TOKEN="ghp_xxx_ou_pat"

printf '%s' "$GHCR_TOKEN" | docker login ghcr.io -u "$GHCR_USER" --password-stdin

Construire, taguer, publier et récupérer le package :

# Construire l'image locale
faas-cli build -f pretty-json.yml

# Publier vers GHCR
faas-cli push -f pretty-json.yml

# Récupérer le package
docker pull ghcr.io/votre-compte/openfaas-pretty-json:1.0.0

Si vous automatisez avec GitHub Actions, utilisez le GITHUB_TOKEN ou un PAT avec les droits write:packages et read:packages selon votre contexte.

Déployer le package dans OpenFaaS

Une fois l'image publiée, OpenFaaS déploie la fonction directement depuis GHCR. Le plus simple est de garder le même tag entre le fichier de fonction et le registre. 

# Déployer la fonction
faas-cli deploy -f pretty-json.yml

# Lister les fonctions
faas-cli list --gateway http://127.0.0.1:8080

# Appeler la fonction
curl -s http://127.0.0.1:8080/function/pretty-json \
  -H "Content-Type: application/json" \
  -d '{"nom":"Ludoc","type":"openfaas"}'

En pratique, le flux propre est : code Python → image Docker → GHCR → déploiement OpenFaaS → invocation HTTP.

Utiliser des fonctions packagées dans OpenFaaS

Le vrai intérêt du packaging apparaît quand vous réutilisez des fonctions déjà packagées. Vous pouvez ainsi :

  • Épingler une version précise pour éviter les régressions
  • Réployer rapidement une fonction dans plusieurs environnements
  • Partager des fonctions entre plusieurs projets sans recoder
  • Mettre à jour un service en changeant uniquement le tag de l'image
# Changer de version
sed -i 's/:1.0.0/:1.0.1/' pretty-json.yml
faas-cli deploy -f pretty-json.yml

# Vérifier l'état
faas-cli describe pretty-json -g http://127.0.0.1:8080

Ce modèle fonctionne particulièrement bien pour des fonctions utilitaires comme le pretty print JSON, des validateurs, des normalisateurs ou des adaptateurs d'API.

Bonnes pratiques

  • Versionnez chaque image avec un tag explicite
  • Utilisez GHCR pour centraliser vos packages Docker
  • Gardez les fonctions courtes et spécialisées
  • Privilégiez des images légères basées sur Python slim
  • Gardez les secrets hors du code et des images
  • Documentez les endpoints et les payloads attendus

Résumé

OpenFaaS est une manière propre d'exposer des fonctions Python, tandis que GHCR fournit la couche de packaging et de distribution. En combinant les deux, vous obtenez un flux moderne : fonction, image Docker, publication, déploiement, invocation. C'est simple, versionnable et facile à automatiser.
