Uptime Kuma - Surveillance de disponibilité moderne et accessible

Uptime Kuma est un outil de monitoring léger, moderne et open-source qui vous permet de surveiller la disponibilité de vos services en temps réel. Découvrez comment le déployer, le configurer et l'intégrer dans votre infrastructure pour garantir une visibilité complète sur l'état de vos applications.

Dernière mise à jour : 09.04.2026

Qu'est-ce qu'Uptime Kuma ?

Uptime Kuma est un outil de surveillance (monitoring) open-source qui monitore la disponibilité (uptime) de vos services, applications et serveurs. Contrairement aux solutions lourdes et complexes, Uptime Kuma propose :

  • Interface web simple et intuitive accessible depuis n'importe où
  • Monitoring multi-protocoles : HTTP, TCP, ICMP (ping), DNS, etc.
  • Alertes en temps réel via email, Slack, Discord, Telegram, etc.
  • Tableaux de bord publics (status pages) pour communiquer avec les utilisateurs
  • Historique et statistiques pour analyser les tendances
  • Déploiement facile en Docker, Kubernetes, ou standalone

Rôles et utilité

Uptime Kuma joue plusieurs rôles importants dans une infrastructure :

  • Surveillance pro-active : détectez les pannes avant vos utilisateurs
  • Gestion d'incidents : réagissez rapidement avec les alertes automatisées
  • Communication de crise : affichez l'état réel via des status pages publiques
  • Analyse de performance : suivez les métriques (temps de réponse, disponibilité %)
  • Conformité et reporting : gardez des traces pour les audits SLA

Installation autonome (recommandée pour démarrer)

La façon la plus simple de commencer : installer Uptime Kuma en tant que service Node.js sur votre machine.

  1. Prérequis : Node.js 14+ et npm installés
  2. Clone et installation : 
git clone https://github.com/louislam/uptime-kuma.git
cd uptime-kuma
npm ci --production
npm run build
npm start

Uptime Kuma sera accessible sur http://localhost:3001. Lors de la première visite, vous serez invité à créer un compte administrateur.

Déploiement Docker (approche moderne et reproductible)

Pour un déploiement en production ou en environnement multi-nœuds, Docker est l'approche idéale. Voici un fichier compose complet et moderne : 

version: '3.8'

services:
  uptime-kuma:
    image: louislam/uptime-kuma:latest
    container_name: uptime-kuma
    restart: always
    volumes:
      - uptime-kuma-data:/app/data
    ports:
      - "3001:3001"
    environment:
      - PUID=1000
      - PGID=1000
    networks:
      - uptime-network
    healthcheck:
      test: ["CMD", "wget", "--quiet", "--tries=1", "--spider", "http://localhost:3001/ping"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s

  # Exemple : monitorer une application locale
  sample-api:
    image: nginx:latest
    container_name: sample-api
    restart: always
    expose:
      - "80"
    networks:
      - uptime-network

volumes:
  uptime-kuma-data:
    driver: local

networks:
  uptime-network:
    driver: bridge

Vous pouvez alors lancer avec : 

docker compose up -d

Configuration d'un premier monitor : HTTP simple

Une fois Uptime Kuma démarré, voici comment configurer votre premier monitor :

  1. Connectez-vous à l'interface (http://localhost:3001)
  2. Cliquez sur "Add New Monitor"
  3. Remplissez :
    • Monitor Type : HTTP(s)
    • Name : "API Ludoc" (exemple)
    • URL : https://api.ludoc.example.com
    • Method : GET
    • Interval : 60 (vérifie toutes les 60 secondes)
    • Timeout : 10 (secondes)
    • Retries : 3 (avant de déclarer "down")
  4. Cliquez sur "Save"

Vous verrez immédiatement un graphique montrant la disponibilité en temps réel.

Types de monitors avancés

Uptime Kuma supporte bien plus que HTTP :

  • TCP : vérifier qu'un port est ouvert (ex: port 5432 pour PostgreSQL)
  • ICMP (Ping) : tester la connectivité réseau brute
  • DNS : vérifier qu'un nom de domaine résout correctement
  • Keyword : HTTP + vérification qu'un mot-clé spécifique est présent en réponse
  • JSON Query : monitorer une API JSON et vérifier une condition
  • MQTT : pour l'IoT et les systèmes temps réel

Configuration des alertes (notifications)

Les alertes vous permettent de réagir immédiatement quand quelque chose tombe en panne. Uptime Kuma supporte de nombreux canaux : 

Canaux supportés :
- Email (SMTP)
- Slack
- Discord
- Telegram
- Webhook personnalisé
- Pushover
- Matrix
- Microsoft Teams
- et bien d'autres...

Pour configurer une notification Slack par exemple :

  1. Allez dans "Settings" → "Notifications"
  2. Cliquez "+ New Notification"
  3. Sélectionnez "Slack"
  4. Collez votre Webhook URL Slack
  5. Testez en cliquant "Test"
  6. Sauvegardez

Puis, dans la configuration de chaque monitor, attachez cette notification pour être alerté en cas de panne.

Status Page publique : communiquer avec l'équipe

Une des forces d'Uptime Kuma : générer des status pages publiques pour informer vos utilisateurs ou votre équipe de la disponibilité réelle.

  1. Allez dans "Status Page"
  2. Cliquez "+ New Status Page"
  3. Donnez un nom : ex "Ludoc Status"
  4. Sélectionnez les monitors à afficher
  5. Configurez l'apparence (titre, logo, couleurs)
  6. Publiez

Vous obtiendrez une URL publique du type http://uptime.example.com:3001/status/ludoc que vous pouvez partager avec votre équipe ou vos clients.

Exemple complet : monitorer une stack application

Voici une configuration réaliste avec plusieurs monitors : 

Monitor 1 : API Backend
- Type: HTTP
- URL: https://api.ludoc.example.com/health
- Expected Keyword: "ok"
- Interval: 30s

Monitor 2 : Base de données
- Type: TCP
- Host: db.ludoc.example.com
- Port: 5432
- Interval: 60s

Monitor 3 : Frontend
- Type: HTTP
- URL: https://ludoc.example.com
- Method: GET
- Interval: 60s

Monitor 4 : Service tiers (Stripe API)
- Type: HTTP
- URL: https://status.stripe.com/
- Interval: 300s (5 minutes)

Notifications configurées :
- Slack pour l'équipe dev (critique)
- Email pour les ops (warning)
- Webhook custom pour le logging centralisé

Bonnes pratiques de configuration

  • Alias descriptifs : nommez vos monitors clairement pour éviter la confusion
  • Intervalles adaptés : pas besoin de checker toutes les secondes (charge inutile), 60s est généralement bon
  • Health checks robustes : utilisez des endpoints dédiés /health plutôt que la racine
  • Multiples notifications : alertez sur plusieurs canaux pour les incidents critiques
  • Sauvegarde régulière : le volume Docker uptime-kuma-data contient tout, assurez-vous qu'il est sauvegardé
  • Évitez les faux positifs : configurez correctement les retries et timeouts

Intégration avec votre infrastructure

Uptime Kuma s'intègre bien avec le reste de votre stack :

  • Avec Docker Swarm : déployez Uptime Kuma en tant que service Swarm et monitrez vos stacks
  • Avec Kubernetes : utilisez l'Helm chart officiel pour un déploiement K8s natif
  • Avec des webhooks : connectez les alertes à votre système de ticketing (Jira, etc.)
  • Avec des PAT Docker : monitrez vos registres privés et pipelines CI/CD

Résumé

Uptime Kuma est un outil puissant et accessible pour surveiller votre infrastructure. Qu'il soit déployé en autonome, en Docker, ou en Kubernetes, il vous permet de gagner en visibilité et de réagir plus rapidement aux incidents. Commencez par quelques monitors critiques, et expandez progressivement avec des alertes et des status pages pour maximiser votre couverture de monitoring.
