Docker PAT - Authentification sécurisée et automatisation CI/CD

Ce cours explique comment utiliser les Personal Access Tokens (PAT) avec Docker Hub pour sécuriser l'authentification, remplacer les mots de passe dans les scripts et fiabiliser vos pipelines. L'objectif : publier, tirer et automatiser des images Docker proprement en environnement de dev et de production.

Dernière mise à jour : 09.04.2026

PAT : c'est quoi ?

Un Personal Access Token est un jeton d'accès personnel lié à votre compte, utilisé comme identifiant machine-à-machine. Contrairement à un mot de passe global, un PAT peut être :

  • scopé sur des permissions ciblées
  • révocable individuellement sans casser tout votre compte
  • rotatif avec une stratégie de renouvellement
  • auditable avec un usage par contexte (CI, local, serveur)

Pourquoi utiliser un PAT avec Docker Hub ?

Le cas Docker est très concret :

  • Push d'images depuis CI/CD
  • Pull d'images privées sur serveur
  • Authentification sur plusieurs machines sans partager le mot de passe principal

Avec Docker, la bonne pratique consiste à utiliser docker login --password-stdin avec un PAT stocké dans un secret manager (GitHub Secrets, GitLab Variables, Vault, etc.).

Créer un PAT Docker Hub

  1. Connectez-vous à Docker Hub.
  2. Ouvrez Account Settings puis Personal access tokens.
  3. Cliquez sur Create token.
  4. Donnez un nom explicite : ci-main-2026q2.
  5. Choisissez les droits minimaux nécessaires.
  6. Copiez le token immédiatement et stockez-le de façon sécurisée.

Évitez les noms génériques comme token1. Un nom clair facilite la rotation et l'audit.

Connexion Docker en local (sans fuite)

Exemple recommandé :

# Linux/macOS
export DOCKERHUB_USERNAME="mon-user"
export DOCKERHUB_PAT="dckr_pat_xxx"

echo "$DOCKERHUB_PAT" | docker login -u "$DOCKERHUB_USERNAME" --password-stdin

Cette approche évite d'exposer le token dans l'historique shell (contrairement à docker login -p ... qui est déconseillé).

Exemple CI/CD GitHub Actions

Dans ce workflow, le PAT est stocké dans DOCKERHUB_TOKEN et injecté au runtime : 

name: Docker Publish

on:
  push:
    branches: ["master"]

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Login Docker Hub
        run: echo "${{ secrets.DOCKERHUB_TOKEN }}" | docker login -u "${{ secrets.DOCKERHUB_USERNAME }}" --password-stdin

      - name: Build image
        run: docker build -t ${{ secrets.DOCKERHUB_USERNAME }}/ludoc:latest .

      - name: Push image
        run: docker push ${{ secrets.DOCKERHUB_USERNAME }}/ludoc:latest

PAT et Docker Compose

Si votre docker-compose.yml tire des images privées, authentifiez d'abord l'hôte (ou l'agent CI) avec le PAT, puis lancez la stack : 

echo "$DOCKERHUB_PAT" | docker login -u "$DOCKERHUB_USERNAME" --password-stdin
docker compose pull
docker compose up -d

L'auth est gérée par le client Docker, vous n'avez pas besoin d'écrire le token dans le fichier Compose.

Rotation et révocation

Stratégie simple et efficace :

  1. Créer un nouveau PAT.
  2. Mettre à jour les secrets CI/CD.
  3. Vérifier les pipelines et déploiements.
  4. Révoquer l'ancien PAT.

Faites cette rotation régulièrement (par exemple tous les 60 à 90 jours) selon votre niveau de risque.

Anti-patterns à éviter

  • Commiter un token dans le code, même en privé
  • Utiliser le même PAT pour tous les environnements
  • Donner des droits trop larges par confort
  • Mettre un PAT en clair dans .env versionné
  • Partager un PAT entre plusieurs personnes

Checklist de mise en production

  • PAT dédié au contexte (CI, serveur, local)
  • Droits minimaux appliqués
  • Stockage dans un coffre/secret manager
  • Rotation planifiée
  • Révocation testée
  • Aucune fuite dans logs et historiques shell

Résumé

Les PAT sont la base d'une authentification Docker moderne : plus sécurisée, plus contrôlable, et mieux adaptée à l'automatisation. En combinant un PAT par contexte, des permissions minimales, et une rotation régulière, vous réduisez fortement le risque de compromission.
