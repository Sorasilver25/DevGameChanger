Husky — Hooks Git automatiques pour lint & tests

Utiliser Husky pour exécuter automatiquement des linters, tests ou scripts au bon moment (pre-commit, pre-push). Installation, automatisation et bonnes pratiques.

Dernière mise à jour : 01.12.2025

1. Quel est le rôle de Husky ?

Husky permet d'ajouter des Git hooks (scripts exécutés lors d'événements Git comme pre-commit, pre-push, commit-msg, etc.) au niveau du projet. Plutôt que d'éditer manuellement les hooks dans .git/hooks, Husky gère des hooks versionnés dans le dépôt (dossier .husky/) et les active automatiquement lors de l'installation des dépendances.

Usage typique :

  • Exécuter un linter (ESLint, Prettier) sur les fichiers staged avant commit.
  • Lancer les tests rapides avant un push.
  • Valider le message de commit (commit-msg).

2. Installation rapide

Exemples d'installation avec npm ou yarn :

# Avec npm
npm install husky --save-dev

# Activer husky (crée le dossier .husky/)
npx husky install

# Ajouter le script 'prepare' dans package.json pour que husky s'installe automatiquement après npm install
# package.json -> "scripts": { "prepare": "husky install" }

# Créer un hook pre-commit qui lance lint-staged
npx husky add .husky/pre-commit "npx --no -- lint-staged"

# Exemple: hook pre-push pour lancer les tests rapides
npx husky add .husky/pre-push "npm test --silent"

Remarque : l'option --no utilisée avec npx évite d'utiliser une version globale si elle existe.

3. Exemple de configuration avec lint-staged

Pour ne linters que les fichiers qui vont être committés, utilisez lint-staged :

# Installation
npm install lint-staged --save-dev

# package.json (extrait)
{
  "scripts": {
    "prepare": "husky install",
    "lint": "eslint . --ext .js,.ts",
    "format": "prettier --write"
  },
  "lint-staged": {
    "*.{js,ts,jsx,tsx}": [
      "eslint --fix",
      "prettier --write",
      "git add"
    ]
  }
}

Avec cette configuration, le hook créé précédemment (.husky/pre-commit) exécutera lint-staged et ne passera le commit que si les règles sont respectées.

4. Exemple de hook .husky/pre-commit

#!/usr/bin/env sh
. "$(dirname "$0")/_/husky.sh"

npx --no -- lint-staged

Le premier bloc (la ligne . "$(dirname "$0")/_/husky.sh") est nécessaire pour charger l'environnement Husky. Le reste du fichier contient la commande à exécuter.

5. Bonnes pratiques

  • Garder les hooks rapides : les hooks ralentissent le workflow si trop lourds. Lancer uniquement des actions rapides (fix, lint, tests unitaires rapides).
  • Utiliser lint-staged : ne linters que les fichiers staged pour un feedback instantané.
  • Préparer un script prepare : ajouter "prepare": "husky install" dans package.json pour que Husky s'active automatiquement après npm install.
  • Ne pas dupliquer la logique CI : Husky est un garde-fou local ; continuez d'avoir la même validation côté CI pour l'assurance qualité.
  • Prévoir des options de contournement : documenter l'utilisation de --no-verify pour des cas exceptionnels et rappeler les conséquences.
  • Versionner les hooks : les scripts dans .husky/ sont versionnés avec le dépôt — évitez d'écrire des hooks non traçables.
  • Logs et messages clairs : afficher des messages d'erreur explicites pour que les développeurs comprennent et corrigent vite.

6. Scénarios courants

  • pre-commit → npx lint-staged (fix + format) : empêche de committer du code mal formaté.
  • pre-push → npm test --silent : détecte les régressions avant push.
  • commit-msg → validation du format du message (Conventional Commits).

7. Commandes utiles — Récapitulatif

# Installer husky et lint-staged
npm install husky lint-staged --save-dev

# Activer husky (ou ajouter dans "prepare" puis npm install)
npx husky install

# Ajouter un pre-commit
npx husky add .husky/pre-commit "npx --no -- lint-staged"

# Ajouter un pre-push
npx husky add .husky/pre-push "npm test --silent"

8. Limitations et alternatives

Husky agit localement et aide à améliorer la qualité du code côté développeur. Cependant :

  • Il ne remplace pas la CI : gardez des contrôles côté serveur.
  • Les hooks peuvent être contournés par --no-verify ; documentez et contrôlez cet usage dans l'équipe.
  • Pour des workflows plus centralisés, envisager des check-runs CI ou des merge checks côté plateforme (GitHub/GitLab).
