RabbitMQ - Message broker, queues et consommateurs

RabbitMQ est un message broker robuste pour découpler les producteurs et les consommateurs. Ce cours explique le rôle d'un broker, l'intérêt concret de RabbitMQ, la notion de queue, de listener et de subscriber, puis montre comment gérer l'environnement et les messages de manière propre.

Dernière mise à jour : 09.04.2026

Qu'est-ce qu'un message broker ?

Un message broker est un intermédiaire entre un producteur et un consommateur. Le producteur envoie un message au broker, puis le broker le route vers une queue, un topic ou un destinataire. Ce modèle évite de coupler directement les services entre eux.

  • Découplage entre émission et traitement
  • Résilience si le consommateur est temporairement indisponible
  • Absorption de charge grâce aux queues
  • Traitements asynchrones et gestion des retries

Pourquoi RabbitMQ ?

RabbitMQ est intéressant quand vous avez besoin d'une messagerie fiable avec des files d'attente, des ack, des dead-letter queues, du routage et plusieurs consommateurs. Il est très utilisé pour :

  • Les jobs en arrière-plan
  • La synchronisation entre microservices
  • Les pipelines d'événements
  • Les traitements batch déclenchés par message

RabbitMQ ou MQTT ?

MQTT est une alternative légère, souvent choisie pour l'IoT, les capteurs et les réseaux contraints. RabbitMQ reste préférable lorsque vous avez besoin d'une vraie logique de broker orientée queues, consommateurs, ack, retry et routage plus riche.

  • MQTT : léger, pub/sub simple, parfait pour les objets connectés
  • RabbitMQ : plus complet pour l'architecture applicative et les workflows d'entreprise

Démarrer RabbitMQ avec Docker

Le plus simple pour tester est d'exécuter RabbitMQ avec le plugin de management. Cela vous donne l'interface web, la visibilité sur les queues et les consommateurs, et un environnement reproductible. 

docker run -d \
  --name rabbitmq \
  -p 5672:5672 \
  -p 15672:15672 \
  -e RABBITMQ_DEFAULT_USER=admin \
  -e RABBITMQ_DEFAULT_PASS=admin123 \
  rabbitmq:3-management

Interface d'administration : http://localhost:15672

Environnement et configuration

En pratique, on externalise les paramètres via des variables d'environnement. Cela permet de garder le code identique entre dev, staging et production. 

export RABBITMQ_HOST=localhost
export RABBITMQ_PORT=5672
export RABBITMQ_USER=admin
export RABBITMQ_PASSWORD=admin123
export RABBITMQ_QUEUE=jobs.pretty-print

Votre application lit ensuite ces variables pour se connecter au broker, publier ou consommer les messages.

Publisher Python

Le publisher envoie des messages dans une queue. Ici, on publie un payload JSON simple vers RabbitMQ. 

import json
import os

import pika


connection = pika.BlockingConnection(
    pika.ConnectionParameters(
        host=os.environ["RABBITMQ_HOST"],
        port=int(os.getenv("RABBITMQ_PORT", "5672")),
        credentials=pika.PlainCredentials(
            os.environ["RABBITMQ_USER"],
            os.environ["RABBITMQ_PASSWORD"],
        ),
    )
)

channel = connection.channel()
queue_name = os.getenv("RABBITMQ_QUEUE", "jobs.pretty-print")
channel.queue_declare(queue=queue_name, durable=True)

payload = {"message": "bonjour", "type": "pretty-print"}
channel.basic_publish(
    exchange="",
    routing_key=queue_name,
    body=json.dumps(payload, ensure_ascii=False),
    properties=pika.BasicProperties(delivery_mode=2),
)

connection.close()

Listener / Subscriber Python

Le listener, aussi appelé subscriber, consomme les messages d'une queue et les traite un par un. L'ack explicite permet de ne valider un message qu'après traitement réussi. 

import json
import os

import pika


def on_message(ch, method, properties, body):
    payload = json.loads(body.decode("utf-8"))
    pretty = json.dumps(payload, indent=2, ensure_ascii=False, sort_keys=True)
    print(pretty)
    ch.basic_ack(delivery_tag=method.delivery_tag)


connection = pika.BlockingConnection(
    pika.ConnectionParameters(
        host=os.environ["RABBITMQ_HOST"],
        port=int(os.getenv("RABBITMQ_PORT", "5672")),
        credentials=pika.PlainCredentials(
            os.environ["RABBITMQ_USER"],
            os.environ["RABBITMQ_PASSWORD"],
        ),
    )
)

channel = connection.channel()
queue_name = os.getenv("RABBITMQ_QUEUE", "jobs.pretty-print")
channel.queue_declare(queue=queue_name, durable=True)
channel.basic_qos(prefetch_count=1)
channel.basic_consume(queue=queue_name, on_message_callback=on_message)
channel.start_consuming()

Queues, ack et dead-letter

Les queues permettent de lisser la charge. Un message peut être traité immédiatement ou mis en attente. Si le traitement échoue, vous pouvez configurer des retries ou une dead-letter queue pour les messages non traités.

  • Queue durable : le message survit au redémarrage du broker
  • Ack manuel : le consommateur confirme après traitement
  • Dead-letter queue : les messages en échec sont isolés
  • Prefetch : contrôle du nombre de messages en cours par consommateur

Exemple d'architecture

Un cas d'usage classique : une API publie un job dans RabbitMQ, puis un worker Python le récupère et exécute un traitement asynchrone. Cela est idéal pour les tâches longues, les emails, les exports ou les notifications. 

API REST -> RabbitMQ queue -> Worker Python -> Base de données / stockage / API tierce

Bonnes pratiques

  • Nommer les queues selon le domaine métier
  • Utiliser des variables d'environnement pour les connexions
  • Rendre les messages idempotents si possible
  • Activer les queues durables et les messages persistants
  • Surveiller la longueur des queues et les consumers actifs
  • Prévoir une dead-letter queue pour les erreurs récurrentes

Résumé

RabbitMQ apporte une vraie couche de messagerie entre vos services : plus de découplage, plus de contrôle, et une gestion plus propre des traitements asynchrones. Si MQTT est une bonne alternative pour l'IoT, RabbitMQ reste très pertinent dès que vous avez besoin de queues, d'ack, de retries et de consommateurs bien définis.
